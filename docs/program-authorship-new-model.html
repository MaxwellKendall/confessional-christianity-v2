Program authorship

The new model is source, review, compile, ship.

A program is still code plus content in git. The change is that prayer authorship now has a human-readable source format and an explicit approval gate before anything reaches the app.

Core rule

Runtime content is generated only from Markdown files whose frontmatter says status: approved. LLM output is always a draft until a human edits and approves it.

The four layers

Program registry

What exists

src/lib/programs.ts defines the program slug, title, catechism, total question count, and estimated time.

Catechism data

What is taught

normalized-data/westminster/wsc.json supplies the questions, answers, and proof-text references.

Authoring source

What is reviewed

content/programs/<slug>/prayers/*.md is the editable source of truth for prayers.

Runtime bundle

What the app imports

prayers.json is compiled from approved Markdown and statically imported by programContent.ts.

The workflow

1
Draft

Create or generate Markdown

Authors can write prayers directly, or use the Anthropic script to draft missing questions from the WSC answer and proof texts.

2
Review

Edit the Markdown file

The file is the editorial surface. A reviewer checks theology, tone, age fit, the {name} placeholder, and the final Amen.

3
Approve

Flip the status gate

The approval command changes status: draft to status: approved. Drafts remain visible in git but invisible to families.

4
Compile

Write the JSON bundle

The compiler reads every prayer file and writes only approved entries into prayers.json. prebuild runs this automatically.

A prayer source file

The frontmatter carries review state and provenance. The body is the prayer that will be compiled once approved.

---
question: 11
status: draft
source: llm
model: claude-sonnet-4-5
generated: 2026-07-11
---
Lord, teach {name} to trust your holy, wise,
and powerful care over all things. Help him
remember that not one ordinary day is outside
your hand. Amen.
Approval boundary

Draft is not unpublished JSON

A draft is not merely hidden by convention. The compile script excludes it, so it cannot reach runtime content unless the frontmatter is approved.

The commands

Generate drafts

Use explicit ranges

npm run prayers:draft -- --questions 11-15 --dry-run
ANTHROPIC_API_KEY=... npm run prayers:draft -- --questions 11-15
Approve reviewed work

Then compile

npm run prayers:approve -- --questions 11-15
npm run prayers:compile
npm run build

What this protects

Families see

Approved, compiled prayers

  • Only entries present in prayers.json.
  • Only text reviewed through git.
  • Missing questions render honestly as unwritten.
Families never see

Draft authoring material

  • LLM output with status: draft.
  • Markdown frontmatter or provenance fields.
  • Any prayer file that has not passed approval.

Mental model

Think of prayers/*.md as the manuscript drawer and prayers.json as the printed booklet. The manuscript drawer can contain drafts, notes, and generated candidates. The booklet contains only approved text, and the app reads only the booklet.
