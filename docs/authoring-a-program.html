Authoring a Program

How a paced, scripture-rooted catechesis plan comes into being — the files you touch, the content you write, and where a human still has to sign off.

A program is an authored, ordered plan that walks a child through a historic catechism one question at a time, with the underlying scripture and a written prayer at each step. There is no admin screen and no database editor: a program is code plus content files, added through a git commit. The review of that commit is the editorial and theological gate.

The steps

Adding a program

  1. Register it in the program list

    Append a ProgramDefinition to the PROGRAMS array. This alone makes the landing page, sitemap entry, and routes exist.

    {
  slug: 'catechizing-heidelberg',
  kind: 'Family Catechesis',
  title: 'Catechizing Your Child: The Heidelberg',
  description: 'A paced walk through the Heidelberg Catechism…',
  catechismId: 'HC',   // see the caveat below
  totalQuestions: 129,
  estimatedMinutes: 15,
}

  2. Author the prayers

    Create content/programs/<slug>/prayers.json — a flat map of question number to prayer text. Use {name} where the child’s name should appear. Keys you omit are treated as unwritten, and the session flow skips them gracefully.

    {
  "_comment": "Written progressively; {name} → the child's name.",
  "1": "Father, you made {name} to know you and be glad in you…",
  "2": "Lord, thank you for not leaving us to guess how to live…"
}

    You do not need all of them on day one. Ten good prayers ship a usable program; the rest arrive in later commits.

  3. Apply the schema (once)

    The account tables that record a child’s progress live in a migration that is written but not yet applied to the live database. Program features error until it runs.

    supabase/migrations/20260710000000_programs_domain.sql

  4. Verify, then commit

    Run the suite and the build, click through the landing and a session locally, then commit the metadata and the prayers together as one change. The commit is the editorial record.

    npm test && npm run build
Before you start

Two constraints to know

Caveat · today only WSC works

The content accessor is getWscQuestion, and catechismId is typed to the literal 'WSC'. A program over the same catechism with different prayers is nearly free; a program over a different catechism (Heidelberg, the Larger Catechism) needs a small refactor first — widen the type and turn getWscQuestion into a catechism-agnostic lookup over the loaders that already exist.

Caveat · identity is the catechism, not the program

A child’s progress rows key off catechism_id, not the program slug. Two programs over one catechism would share a single progress record and collide. If you want that, the schema needs a program_slug — and since the migration isn’t applied yet, now is the cheap moment to change it.

Optional

Drafting prayers with an LLM

Because every question already carries its answer and proof texts, an LLM can draft the prayers for you — with the existing prayers as style anchors. Treat it as a drafting accelerant, never as autopublish: generated prayers land as unreviewed drafts, and a human reads, edits, and approves each before it reaches a family. Your commit workflow already is that review gate.

Proposed · not yet built

A clean version of this would author prayers as per-file Markdown (matching the commentary system), keep an approved vs draft split, and add a scripts/ step that compiles only approved prayers down to the bundled prayers.json the app imports. This separates authoring ergonomics from the runtime format and keeps unreviewed text out of the product.

The shape of a generation run

  • For each question, prompt with its Q, A, and proof-text refs, plus the shipped prayers as few-shot examples.
  • Constrain hard: Reformed framing, address the child by {name}, two to three sentences, lean on the proof texts, close with “Amen.”
  • Write results as drafts, flagged unreviewed.
  • A human approves; only approved prayers compile into the app.
